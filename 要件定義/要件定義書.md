## 1. はじめに

### 1.1 文書の目的  
本書は、株式会社〇〇が構築するオンライン販売サイト（ECサイト）に関するシステム要件を明確にし、開発チーム・クライアント双方の共通理解を図ることを目的とする。  

### 1.2 背景
- 株式会社〇〇はこれまで実店舗を中心に雑貨（文房具、キッチン小物など）の販売を行ってきたが、販路は地域に限定されていた。  
- SNS（Instagram等）を活用して商品紹介や個別問い合わせ対応による販売も行っているが、受注処理や問い合わせ対応に手間がかかり、業務の効率性に課題がある。  
- 競合他社がオンライン販売チャネルを拡大しており、遠方の潜在顧客からも購買希望が寄せられている。
 
### 1.3 目的
- 地域外の顧客にも商品を届けるため、オンライン販売チャネルを確立する。  
- シンプルで使いやすく商品の魅力が伝わるECサイトを構築する。  
- SNSによる個人対応の非効率を解消し効率的な販売運営を目指す。

## 2. システム概要

-  システム名  :   株式会社〇〇 オンライン販売サイト（ECサイト）                                                  
- 対象ユーザー:管理者、購入者
- 利用環境 : PCおよびスマートフォン（iOS/Android）、主要ブラウザ対応 
                                  

---

## 3. 業務要件

###  ユーザー要件  
- 誰でも簡単に商品を閲覧し、購入できる。  
- 商品写真を中心に魅力が伝わるUIで操作性が高いこと。  
- 会員登録は任意とし、未登録でも購入可能。

## 5. 非機能要件（Non-functional Requirements）

### 5.1 UI/UX・デザイン要件  
- 白を基調としたシンプルで清潔感のあるデザイン。  
- 商品写真が美しく映えるレイアウト設計。  
- 直感的な操作導線で、初めてのユーザでも迷わず操作可能。  
- フォント、ボタン配置、余白に配慮し視認性を高める。  

### 5.2 スマートフォン対応  
- レスポンシブデザインを採用し、iOS・Androidの標準ブラウザ（Safari、Chrome）での正常動作を保証。  
- タッチ操作に適した大きめのボタンや入力フォームを設計。  
- 画面サイズに応じてレイアウトを最適化。  

### 5.3 パフォーマンス・可用性  
- 初期リリースは小規模トラフィックを想定。  
- ページ表示速度は2秒以内を目標。画像の最適化やキャッシュ利用を検討。  
- サイトの稼働率は99.5%以上を目指す。  

### 5.4 拡張性・保守性  
- 将来の機能追加（決済、レビュー、会員機能など）に対応しやすいモジュール設計。  
- 商品カテゴリや商品数の増加に柔軟に対応可能なデータ構造。  
- 管理画面の機能拡張を見据えた設計。  

### 5.5 セキュリティ  
- SSL/TLS通信によるデータ暗号化を必須とする。  
- 入力フォームのバリデーション（メール形式、必須項目など）を徹底。  
- 管理画面はID・パスワードによる認証制限を設け、不正アクセスを防止。  
- 個人情報保護法に準拠したデータ管理と利用規約・プライバシーポリシーの整備。

## 7. スコープと対象外事項

### 7.1 スコープ(開発対象範囲)

-本要件定義書「4.機能要件」に記載された機能の開発・実装
-本要件定義書「5.非機能要件」を満たすシステムの構築
-指定された利用環境での動作保証
-受入テストの実施支援


### 7.2 対象外事項(開発対象外範囲)

-初回リリースでの決済機能実装
-初回リリースでのレビュー機能
-複雑な在庫管理機能
-管理者向け機能(管理画面)
-カテゴリ別表示機能
-商品検索機能
-会員情報の編集
-登録会員の購入履歴閲覧

## 8. 用語集

| 用語 | 説明 |
|------|------|
| **ECサイト** | Electronic Commerce（電子商取引）サイトの略。Web上で商品やサービスの販売・購入を行うサイト。 |
| **チャネル（販売チャネル）** | 商品やサービスを顧客に届ける手段・経路。例：実店舗、オンラインショップ、SNSなど。 |
| **潜在顧客** | 現時点では購入していないが、将来的に顧客になる可能性のある見込み顧客。 |
| **購入者（ユーザー）** | ECサイトを利用し、商品を閲覧・購入する一般ユーザー。会員登録の有無は問わない。 |
| **管理者** | サイト運営を担当し、商品・注文・顧客情報などを管理するユーザー。 |
| **商品カテゴリ** | 商品を分類するための区分。例：文房具、キッチン用品など。将来変更の可能性がある。 |
| **サムネイル画像** | 商品一覧などに表示される小型の商品画像。クリックで詳細ページへ移動。 |
| **UI（ユーザーインターフェース）** | ユーザーがシステムとやり取りするための操作画面やデザインのこと。視認性や操作性が重視される。 |
| **レスポンシブUI / レスポンシブデザイン** | 画面サイズやデバイスに応じてレイアウトを調整するデザイン方式。PC・スマホ両方で最適に表示される。 |
| **UX（ユーザーエクスペリエンス）** | ユーザーがシステム利用を通じて得る体験のこと。使いやすさ・満足度に関わる重要な指標。 |
| **カート機能** | 購入予定の商品を一時保存・管理する機能。数量変更や削除も可能。 |
| **注文ステータス** | 注文処理の進捗状況を示す分類。例：「受注済み」「出荷済み」など。 |
| **非会員購入** | 会員登録せずに商品購入が可能な仕組み。購入ハードルを下げることができる。 |
| **会員登録機能** | 氏名・住所・メールなどを登録し、次回購入時の利便性（情報自動入力、履歴閲覧など）を提供する機能。 |
| **受注処理** | ユーザーからの注文を管理者が確認し、ステータス変更や出荷準備を行う業務フロー。 |
| **機能要件** | システムが実現すべき「具体的な機能」を定義したもの。例：商品表示、注文機能、管理画面など。 |
| **非機能要件** | 機能以外の品質的要求。例：パフォーマンス、セキュリティ、操作性、拡張性など。 |
| **トラフィック** | Webサイトへのアクセス量やデータ通信の総量。多くなると処理性能が求められる。 |
| **画像の最適化** | ページ表示速度向上のため、画像サイズやフォーマットを最適に調整する処理。 |
| **キャッシュ利用** | よく使うデータ（画像・HTML等）を一時保存して、再表示時の速度を向上させる技術。 |
| **モジュール設計** | システムを再利用可能な部品（モジュール）単位で構成する設計手法。拡張性や保守性が高まる。 |
| **スコープ** | プロジェクトで対応する開発・実装・テストなどの範囲。対象外事項との明確な区別が必要。 |
| **入力バリデーション（バリデーション）** | 入力内容（例：メール形式、必須項目）に誤りがないかチェックする機能。セキュリティやUXの向上に必須。 |
| **プライバシーポリシー** | ユーザーの個人情報の収集・利用・管理方針を明記した文書。法的義務に基づき策定される。 |
| **SSL/TLS** | Web通信を暗号化するプロトコル。個人情報や決済情報などを安全に送受信するために使用。 |
| **フレームワーク** | Webアプリケーション開発の効率を上げるための開発基盤。例：Laravel、Django、Reactなど。 |
| **稼働率** | システムが正常に利用可能な時間の割合。例：稼働率99.5%＝年間の停止時間が約44時間以内。 |
